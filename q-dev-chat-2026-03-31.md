Great security question. Here's the full picture:

---

**The problem with public endpoints:**

If you expose `POST /api/chat?agentId=xyz` without any auth, anyone can:
- Spam your endpoint → burn your Mistral API credits
- Scrape your agent's knowledge base
- Use your Railway server for free

---

**How production embed widgets solve this (standard approach):**

**1. Widget API Key — not JWT**

Instead of JWT (user login token), you generate a **widget-specific API key** per agent:
```
wk_live_abc123xyz...
```

This key is:
- Public (embedded in the script tag — yes, visible to anyone)
- Scoped to ONE agent only
- Can be revoked anytime
- Rate limited (e.g. 100 requests/day per domain)

```html
<script>
  window.ArchelonConfig = {
    widgetKey: "wk_live_abc123xyz",  ← public, scoped key
  };
</script>
```

**2. Domain whitelist**

When user registers their widget, they add allowed domains:
```
allowed_domains: ["mywebsite.com", "app.mywebsite.com"]
```

Backend checks `Origin` header on every widget request — if domain not in whitelist, reject. This means even if someone steals the widget key, they can't use it from their own domain.

**3. Rate limiting**

Per widget key, per day limit. If exceeded → 429 error. Protects your Mistral credits.

---

**What about Railway URL?**

You don't expose Railway URL directly. You put **Cloudflare** or a **custom domain** in front:

```
widget request → api.archelon.ai → Railway backend
```

User never sees `archelon-production.up.railway.app`. They only see `api.archelon.ai`.

---

**So the security stack for widget:**

```
Widget Key (scoped) + Domain Whitelist + Rate Limit + Custom Domain
```

This is exactly how Intercom, Crisp, Tidio all work. Standard, safe, production-ready.
