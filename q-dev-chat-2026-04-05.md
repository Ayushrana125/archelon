How did you added Rate Limiter? In code you did not server?

---

Good question. I added it **in code** — pure Python, no server-level tool.

Here's exactly what it does:

```python
_rate_store: dict[str, list[float]] = defaultdict(list)

def check_rate_limit(user_id: str, limit: int = 20, window: int = 60):
    now = time.time()
    # Keep only timestamps within the last 60 seconds
    _rate_store[user_id] = [t for t in _rate_store[user_id] if now - t < window]
    # If 20 or more requests in that window — block
    if len(_rate_store[user_id]) >= limit:
        raise HTTPException(status_code=429, detail="Too many requests. Please slow down.")
    # Otherwise record this request's timestamp
    _rate_store[user_id].append(now)
```

It's a **sliding window counter** stored in memory. For each user, it keeps a list of timestamps of their recent requests. On every request it:
1. Removes timestamps older than 60 seconds
2. Counts what's left
3. If 20+ → reject with 429
4. Otherwise → add current timestamp and allow

---

**The limitation of in-code rate limiting:**

`_rate_store` is a Python dict that lives in RAM. This means:

- **Railway restart** → dict is wiped → rate limit resets. Someone could abuse this.
- **Multiple Railway instances** (replicas) → each has its own dict → a user could hit 20 req on instance 1, then 20 more on instance 2. Total 40 before being blocked.

**How real companies solve this:**

They use **Redis** — a shared in-memory store outside the app. All instances read/write the same counter. Survives restarts.

**For your current scale** — one Railway instance, no replicas — the in-code approach is completely fine. You'd only need Redis when you scale to multiple replicas, which is a Hobby/Pro tier decision you'd make much later.

So the answer is: **code-level rate limiting, not server-level**. Server-level would be something like Nginx or Cloudflare rate limiting at the network layer before requests even reach your Python code.
