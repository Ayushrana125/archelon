Use `mistral-embed` — it's their dedicated embedding model.
- **Max tokens per request**: 16,384 tokens
- **Rate limits (free tier)**: 1 req/sec, 500K tokens/min
- **Rate limits (paid tier)**: much higher, ~10 req/sec

### Batching Strategy

Your child chunks are 8–150 tokens each. The goal is to **maximize tokens per API call** to minimize total calls.

**Simple math for your example:**
- 300 child chunks, 2000 total tokens
- Max 16,384 tokens per call
- → **1 single API call** handles all 300 chunks

For a heavier doc:
- 500 chunks × 150 tokens avg = 75,000 tokens
- → ceil(75,000 / 16,384) = **5 API calls**

So batch by **token budget, not chunk count**. Fill each batch up to ~15,000 tokens (leave buffer), then send.

---

### Time Estimate

Ingestion took 1.5–2.5s. Embeddings will be different:

| Scenario | Chunks | Batches | Est. Time |
|---|---|---|---|
| Small doc (your 300 chunks) | 300 | 1 call | ~1–2s |
| Medium doc (500 chunks) | 500 | 5 calls | ~5–10s |
| Large doc (1000 chunks) | 1000 | 10 calls | ~10–20s |

With async + rate limiting between calls, realistic estimate is **2–15 seconds** per document depending on size.

---

### Async vs Sync

You **can and should** do async here. The pattern:

```python
# Process batches sequentially (not parallel) to respect rate limits
for batch in batches:
    response = await embed_batch(batch)
    await asyncio.sleep(0.5)  # 500ms between calls is safe on paid tier
```

**Do NOT** fire all batches in parallel — you'll hit rate limits instantly.

---

### Rate Limit Strategy

- **Free tier**: Keep 1–1.5s between calls. Max 1 req/sec.
- **Paid tier**: 0.3–0.5s between calls is safe.
- Your 20–25s was very conservative — for async you can go much tighter.

**One key is enough** if you're on paid tier. Multiple keys from the same account don't help — rate limits are account-level, not key-level. 1.4  Error Handling — try / except
Things go wrong in production. APIs fail. JSON is malformed. Without error handling, your program crashes completely. With it, you control what happens instead of crashing.

1.8  Async / Await
This is one of the most important concepts for AI development. LLM API calls take 3-8 seconds. Without async, every user waits behind every other user. With async, many users are served simultaneously.
# SYNCHRONOUS — blocking
def get_response():
    result = call_llm()    # takes 5 seconds, nothing else happens
    return result          # User B waits. User C waits. Everyone waits.
 
# ASYNCHRONOUS — non-blocking
async def get_response():
    result = await call_llm()  # waiting, but other requests can run
    return result              # User B gets served while User A waits for LLM
 
# Rules:
# async def  — marks a function as asynchronous (a &#39;coroutine&#39;)
# await      — pause HERE and wait, but let other things run
# await can only be used INSIDE an async def function
 
# Real FastAPI pattern:
async def chat(request: ChatRequest):
    response = await llm.ainvoke(prompt)   # non-blocking LLM call
    return {&#39;answer&#39;: response}

💬 Interview framing
FastAPI is async by default — that&#39;s the primary reason I chose it over Flask for Archelon. Every chat request involves an LLM call that takes several seconds. Async means while one user&#39;s request is processing, others are being served simultaneously.

1.9  Concurrency — Threading vs Multiprocessing vs Async
This is the one that trips everyone up. Three ways to do multiple things at once in Python, and they work completely differently. You&#39;ve used ThreadPoolExecutor in your Localization Engine — here&#39;s the full picture.

Approach	Best For	You&#39;ve Used It
Threading	IO-bound tasks — waiting for APIs, files, network	Localization Engine — 4 workers calling Mistral API simultaneously
Multiprocessing	CPU-bound tasks — heavy computation, image processing	Rarely in your current work — for number crunching
Async / Await	IO-bound tasks — same as threading but lighter weight	FastAPI, LangChain async calls

The GIL — Why Threading Has a Catch
Python has a Global Interpreter Lock (GIL). It means only ONE thread can execute Python code at a time. But — when a thread is waiting for a network response (like an API call), it releases the GIL. So other threads can run during that wait.
This is why Threading works for your Localization Engine — the 4 workers spend most of their time waiting for Mistral to respond. During that wait, other workers run. If you were doing heavy number crunching instead, Threading wouldn&#39;t help — use Multiprocessing then.
from concurrent.futures import ThreadPoolExecutor
 
def call_api(row):
    return requests.post(url, json=row)   # waits for network — releases GIL
 
with ThreadPoolExecutor(max_workers=4) as executor:
    results = list(executor.map(call_api, rows))
    # 4 calls happen &#39;simultaneously&#39; — each waits for API response
    # while waiting, others run → ~4x throughput for IO-bound work

1.10  Generators and yield
A generator is a function that produces values one at a time instead of all at once. This is how streaming works — whether it&#39;s LLM token streaming or processing a huge file without loading it all into memory.
# Normal function — creates the whole list in memory first
def get_numbers():
    return [1, 2, 3, 4, 5]    # all 5 in RAM at once
 
# Generator function — produces one value at a time
def get_numbers_lazy():
    yield 1    # pause here, give 1, wait for next()
    yield 2    # resume, give 2
    yield 3    # ...
 
gen = get_numbers_lazy()
next(gen)    # → 1
next(gen)    # → 2
 
# More common — generator expression (like list comprehension but lazy)
squares = (x*x for x in range(1000000))  # no RAM used yet
next(squares)   # calculates ONLY the first one
 
# Real use — streaming LLM tokens in FastAPI
async def generate():
    async for chunk in llm.astream(prompt):
        yield chunk.content   # send one token at a time to the client

Where you&#39;ve used this
Your FastAPI streaming endpoint is a generator. Each yield sends one chunk of the LLM response to the frontend. The user sees text appearing word by word because your generator produces chunks as they arrive from Mistral.

1.11  Context Managers — the with statement
The with statement is how Python handles things that need to be opened and closed — files, database connections, network sessions. It guarantees cleanup even if an error happens.
# Without with — you must remember to close()
f = open(&#39;data.csv&#39;, &#39;r&#39;)
content = f.read()         # if this crashes, f never gets closed
f.close()                  # might never run
 
# With context manager — guaranteed cleanup

2.2  Stack and Queue
These are patterns for how you access data. You don&#39;t need a special class — Python lists and deque handle both.
Stack — Last In, First Out (LIFO)

2.3  Big O Notation — Just the Concept
Big O describes how a function&#39;s performance scales as input grows. You don&#39;t need to calculate it mathematically. You need to know what the common ones mean and recognize them.
Notation	Name	What it means in plain English
O(1)	Constant	Same speed regardless of size. dict lookup, set membership. Looking up a value by key.
O(log n)	Logarithmic	Gets slower very slowly as size grows. Binary search on sorted data.
O(n)	Linear	Doubles when size doubles. Scanning every item in a list.
O(n log n)	Linearithmic	Slightly worse than linear. Good sorting algorithms — merge sort, Python&#39;s sorted().
O(n²)	Quadratic	Squares when size doubles. Nested loop over same data. Avoid for large datasets.

Where this matters for you
When you search a list with &#39;item in list&#39;, that&#39;s O(n) — it scans every element. When you search a dict or set with &#39;item in dict&#39;, that&#39;s O(1) — instant. When you cleaned 6.5 lakh records, using sets for deduplication instead of nested loops was the difference between seconds and hours.

2.4  Search and Sort — Conceptual
Linear Search — O(n)
Scan every element until you find it. Simple but slow for large data.
def linear_search(data, target):
    for i, item in enumerate(data):
        if item == target:
            return i    # found at index i
    return -1           # not found

Binary Search — O(log n)
Only works on sorted data. Divide and conquer — check the middle, eliminate half the remaining data each time. Finding a word in a dictionary — you don&#39;t read every word, you open the middle and go left or right.
# Python has this built in:

# Concept (don&#39;t need to memorize implementation):
# 1. Check middle element
# 2. If target == middle: found
# 3. If target &lt; middle: search left half
# 4. If target &gt; middle: search right half
# 5. Repeat — eliminates half the data each step

2.5  Recursion — Functions That Call Themselves
A recursive function calls itself with a smaller version of the problem until it reaches a base case. The base case stops the recursion — without it, you get infinite recursion and a crash.
def countdown(n):
    if n &lt;= 0:          # BASE CASE — stop here
        print(&#39;Done&#39;)
        return
    print(n)
    countdown(n - 1)    # RECURSIVE CALL — smaller problem
 
countdown(3)  # prints: 3, 2, 1, Done
 
# Classic example — factorial
def factorial(n):
    if n &lt;= 1:          # base case
        return 1
    return n * factorial(n - 1)  # recursive call
 
factorial(5)  # 5 * 4 * 3 * 2 * 1 = 120

Where you&#39;ve touched this

3.5  CORS — Why Browsers Block Your API

3.6  HTTP vs HTTPS
HTTP is plaintext — anyone watching the network can read the data. HTTPS is HTTP with TLS encryption — data is encrypted in transit. The S stands for Secure.
TLS (Transport Layer Security) works by: server has a certificate from a trusted authority (Let&#39;s Encrypt in your case), client verifies it, they negotiate an encryption key, all communication is encrypted with that key.
You set this up with Nginx and Let&#39;s Encrypt on your VPS. Nginx handles the TLS termination — it receives encrypted HTTPS traffic, decrypts it, and forwards plain HTTP to your app running on localhost. Your app never sees the encryption complexity.

3.8  WebSockets and Server-Sent Events
Server-Sent Events (SSE) — One-Way Streaming
Standard HTTP is request-response — client sends a request, server sends one response. SSE keeps the connection open and lets the server push data continuously. This is how your Archelon chat streaming works — the server sends LLM tokens one by one as they arrive.
# FastAPI streaming response — this is SSE under the hood
from fastapi.responses import StreamingResponse
 
@app.post(&#39;/chat&#39;)
async def chat(request: ChatRequest):
    async def generate():
        async for chunk in llm.astream(prompt):
            yield f&#39;data: {chunk.content}\n\n&#39;  # SSE format
    return StreamingResponse(generate(), media_type=&#39;text/event-stream&#39;)

WebSockets — Two-Way Real-Time
WebSockets create a persistent two-way connection. Both client and server can send messages at any time without a new request. Supabase Realtime uses WebSockets — when a row changes in your database, it pushes the update to all connected clients instantly.
# WebSocket in FastAPI:
from fastapi import WebSocket
 
@app.websocket(&#39;/ws&#39;)
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept()          # handshake
    while True:
        data = await websocket.receive_text()   # wait for message
        await websocket.send_text(f&#39;Echo: {data}&#39;)  # reply

Connection	New connection per request	Stays open, server pushes	Persistent two-way connection

### What is CORS?

CORS stands for **Cross-Origin Resource Sharing**.

To understand it, you first need to understand what an "origin" is.

An **origin** is the combination of:
- Protocol (`http` or `https`)
- Domain (`localhost`, `archelon.vercel.app`)
- Port (`5173`, `8080`)

So `http://localhost:5173` is one origin. `https://archelon-production.up.railway.app` is a completely different origin.

---

### The Browser's Security Rule

Browsers have a built-in security rule:

> **"You cannot make a request from one origin to a different origin unless the server explicitly allows it."**

This is called the **Same-Origin Policy**.

Why does this exist? Imagine you're logged into your bank. A malicious website you visit tries to silently make a request to your bank's API using your cookies. The browser blocks this — because the malicious site's origin is different from the bank's origin.

---

### So What Happens in Your Case?

Your frontend runs on `http://localhost:5173` (or `https://archelon.vercel.app`).

Your backend runs on `https://archelon-production.up.railway.app`.

These are **different origins**. So when your frontend tries to call the backend, the browser first asks the backend:

> "Hey, do you allow requests from `localhost:5173`?"

This is called a **preflight request**.

If the backend says yes by sending back the right headers, the browser allows the request. If the backend crashes with a 500 before it can respond — it sends back **no headers at all**, and the browser blocks the request entirely and shows:

```
Access to fetch has been blocked by CORS policy: 
No 'Access-Control-Allow-Origin' header is present
```

---

### Why Did You See This Error?

Your backend was **crashing with a 500** before FastAPI could even attach the CORS headers to the response. The crash happened in `get_ingestion_job` because `.single()` threw an exception when the job row was deleted.

FastAPI's CORS middleware adds headers to **successful responses**. When the app crashes hard enough, the response goes out raw with no headers — browser blocks it.

---

### How You Fixed It in FastAPI

In your `main.py` you already have:

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_methods=["*"],
    allow_headers=["*"],
)
```

This tells the browser: **"This server accepts requests from any origin."**

`allow_origins=["*"]` means all origins are allowed. In production you'd normally restrict this to just your frontend domain, but `*` is fine for now.

---

### Summary in One Line

CORS is the browser asking the server "are you okay with me talking to you?" — and the server crashing before it can answer causes the browser to block the request entirely.

What is 512 inputs? And how tokens are actually calculated?

---


Good questions. Let me explain both.

---

### What is 512 inputs?

An "input" in this context is simply **one chunk of text** you send to the embedding API.

When you call Mistral's embedding API, you send a list of texts like:

```python
{
    "model": "mistral-embed",
    "input": [
        "Python is a programming language...",   # input 1
        "A function is a reusable block...",      # input 2
        "HTTP stands for HyperText...",           # input 3
        ...                                       # up to 512
    ]
}
```

Mistral says: **"I will process maximum 512 texts in one API call."**

Your doc had many small chunks — each chunk was only 10-30 tokens, so the total token count was well under 15,000. But the number of chunks was over 512. So our token-based batching said "one batch is fine" but Mistral rejected it saying "too many texts in one request."

Think of it like a restaurant — you can order food worth ₹5000 in one order (token limit), but you can only order maximum 10 dishes at a time (input limit). Even if each dish costs ₹50, you can't order 200 dishes at once.

---

### How Tokens Are Actually Calculated

A **token** is not a word. It's a chunk of characters that the model's tokenizer splits text into.

The general rule of thumb:
- **1 token ≈ 4 characters** in English
- **1 token ≈ 0.75 words** roughly

Examples:
```
"Hello"          → 1 token
"Hello world"    → 2 tokens
"Ayush"          → 1 token  
"Unbelievable"   → 3 tokens  (Un-believ-able)
"API"            → 1 token
"def greet():"   → 4 tokens
```

Special characters, code symbols, and non-English text often use **more tokens per character** because the tokenizer wasn't trained on them as much.

---

### Why Your Estimator Was Off

In your chunker you have:

```python
def estimate_tokens(text: str) -> int:
    return max(1, len(text) // 4)
```

This says: **token count = number of characters divided by 4**.

This is a rough approximation. It works okay for plain English prose but **underestimates badly for code** because:

- Code has lots of symbols like `{`, `}`, `(`, `)`, `->`, `**` — each symbol is often its own token
- Indentation spaces count as characters but barely as tokens
- Variable names like `batch_insert_parent_chunks` might be 1-3 tokens but is 30 characters

So your chunker thought a code block had 100 tokens when it actually had 300+. That's why chunks seemed small but the actual API was getting overwhelmed.

---

### The Accurate Way

The accurate way is to use the actual tokenizer library:

```python
from mistral_common.tokens.tokenizers.mistral import MistralTokenizer

tokenizer = MistralTokenizer.v3()

def count_tokens(text: str) -> int:
    tokens = tokenizer.encode(text)
    return len(tokens.tokens)
```

But this adds a dependency and slows down chunking. For your current scale, the `// 4` estimate + the 512 input cap we just added is good enough.

### Is 118 seconds safe in production?

**No, not really.** Here's why:

Railway's default request timeout is **30 seconds**. But your ingestion runs as a `BackgroundTask` — meaning the HTTP request returns immediately with the job IDs, and the actual processing happens in the background. So the 118 seconds doesn't hit any request timeout. That part is fine.

**The real risks are:**

1. **Railway's container can be restarted** — Railway restarts containers during deploys, scaling events, or crashes. If a restart happens mid-ingestion, the background task dies silently. The job stays stuck in `embedding` status forever with no error.

2. **Supabase connection pool exhaustion** — The Supabase Python client uses a synchronous HTTP client under the hood. Running 20 concurrent `asyncio.gather` calls against it means 20 simultaneous connections. The sync client isn't designed for this — it could deadlock or throw connection errors.

3. **Memory** — 271 embedding vectors, each 1024 floats × 4 bytes = ~1MB held in memory during the update loop. Fine for now, but worth knowing.

## What I'd Refine

### 1. "Spawning multiple retrieval agents as a team" — simplify this

### 2. Neighbour chunks — be specific about what you mean

There are two approaches:

**Option A — Parent chunk expansion**
You search child chunks (small, precise), then fetch their parent chunk (larger context). This is called "small-to-big retrieval" and it's the standard approach. You already have this structure in your DB.

**Option B — Adjacent chunk retrieval**
Fetch the chunk before and after the matched chunk by `chunk_index`. Useful when an answer spans two consecutive chunks.

I'd recommend **Option A** for now — it's simpler and your parent/child structure is already built for it.

### 3. Re-ranking — don't skip this

You mentioned derank. This is important. After retrieval you'll have chunks sorted by vector similarity score. But similarity score isn't the same as relevance to the actual question. A cross-encoder re-ranker (like `flashrank` — free, runs locally) re-scores chunks by actually reading the query + chunk together. Top 3-5 after re-ranking are much better than top 3-5 by raw similarity.

- Single query answer: 300-500 tokens
- Multi query answer: 800-1200 tokens

1500 tokens of context for a single query means roughly 10-15 child chunks. That's actually quite generous. The question is quality over quantity — 5 highly relevant chunks beat 15 loosely relevant ones.

- Retrieve top 20 by vector similarity
- Re-rank
- Take top 5
- Expand to parent chunks
