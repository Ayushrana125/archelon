**Archelon ‚Äî Backend + Frontend Documentation**
Date: April 2026
Author: Ayush Rana

## Why Streaming Was Needed

Before streaming, the full pipeline ran sequentially and the frontend received one complete JSON response at the end:

```
classify (1s) ‚Üí analyse (0.8s) ‚Üí vector search (0.5s) ‚Üí Mistral generates full answer (2-4s) ‚Üí sendd
```

User stared at a spinner for 4-6 seconds before anything appeared. Perceived wait time was 10-12 seconds....

With streaming, the first token appears ~2.5 seconds after the user sends a message. Words appear as Mistral generates them. Perceived wait dropped to 3-5 seconds..

---

## How SSE Streaming Works

The backend keeps the HTTP connection open and pushes small chunks (tokens) as they are generated by Mistral, using **Server-Sent Events (SSE)** format:

```
data: {"type": "meta", "intent": "single", "thinking": "...", "search_queries": [...]}

data: {"type": "token", "token": "Archelon"}
data: {"type": "token", "token": " is"}
data: {"type": "token", "token": " an"}
...
data: {"type": "done", "sources": [...], "token_usage": {...}}
```

Three event types:
- `meta` ‚Äî sent first, carries intent + thinking + search_queries so frontend can render ThinkingSteps immediately
- `token` ‚Äî one per Mistral output token, appended live to the message bubble
- `done` ‚Äî carries sources and token usage, triggers commit of streaming bubble to messages

## Backend Changes

### `pipeline/synthesizer.py`

Added `synthesize_stream()` ‚Äî an async generator that streams tokens instead of returning a complete response.

Key change: switched from `llm.ainvoke()` to `llm.astream()`:

```python
# Before
response = await llm.ainvoke([...])
return response.content

# After
async for chunk in llm.astream([...]):
    yield f"data: {json.dumps({'token': chunk.content})}\n\n"
```

Also extracted `_build_system_prompt()` as a shared helper used by both `synthesize()` (non-streaming, kept for compatibility) and `synthesize_stream()`.

### `routers/chat.py`

Added `POST /api/chat/stream` endpoint using FastAPI's `StreamingResponse`.

Pipeline flow inside the endpoint:
1. Rate limit check
2. Token balance check (402 if exhausted)
3. `classify_and_analyse()` ‚Äî single LLM call for intent + search queries
4. If smalltalk ‚Üí stream answer directly, no retrieval
5. If single/multi ‚Üí vector search (parallel fan-out) ‚Üí rerank ‚Üí `synthesize_stream()`
6. Token logging happens inside the stream after `done` event

Two headers required for Railway/nginx to not buffer the stream:
```python
headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"}
```

## Frontend Changes

### `ChatView.jsx`

Replaced `fetch` + `await res.json()` with a `ReadableStream` reader:

```javascript
const reader = res.body.getReader();
const decoder = new TextDecoder();
let buffer = '';

while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    // parse SSE lines from buffer
}
```

Event handling:
- `meta` ‚Üí adds ThinkingSteps message to chat, clears isTyping spinner
- `token` ‚Üí appends to `streamingMsg.content` live ‚Äî ReactMarkdown re-renders on each token
- `done` ‚Üí updates sources, logs tokens, commits streaming bubble to messages after 120ms

Removed `TypewriterMessage` for RAG responses ‚Äî real tokens from Mistral replace the fake word-by-word animation. `TypewriterMessage` is kept only for smalltalk responses.

### Smalltalk Streaming

Smalltalk sends the full answer as one token (since `handle_smalltalk` returns a complete string). To avoid showing the full text then re-animating it:

- Smalltalk streaming bubble is marked `hidden: true` ‚Äî accumulates content invisibly
- Shows only a spinning logo placeholder while loading
- On `done` event, commits as `role: 'smalltalk'` ‚Äî `TypewriterMessage` animates from empty

### ThinkingSteps Auto-Collapse

Added `collapseNow` prop to `ThinkingSteps`. When `streamingMsg` becomes truthy (first token arrives), `collapseNow={true}` fires and the thinking panel smoothly collapses. Answer streams in below.

`isHistorical` fixed to only be `true` when an assistant message already exists after the thinking message ‚Äî not during live queries.

## Performance Impact

| Metric | Before | After |
|---|---|---|
| Perceived response time | 10-12 seconds | 3-5 seconds |
| Time to first word | 4-6 seconds | ~2.5 seconds |
| Actual Mistral generation time | unchanged | unchanged |

The improvement is entirely perceptual ‚Äî streaming doesn't make Mistral faster, it makes the wait invisible by showing progress immediately.
