`summary_embedding` on `parent_chunks` is interesting — you're planning to embed parent chunk summaries too, not just child chunks. This enables a two-stage retrieval:
1. Search parent summaries first (broad match)
2. Then search child chunks within matched parents (precise match)

That's called **hierarchical retrieval** — more advanced than what most RAG tutorials show. Good forward thinking, but build child-only retrieval first and add this later.

```sql
ALTER TABLE child_chunks ADD COLUMN agent_id UUID REFERENCES agents(id);
```

Great question. This is called **layered architecture**. Here's how it works and how to organise it:

---

## The Layers

```
Frontend
    ↓ HTTP
main.py          — Routes only, no logic
    ↓
routers/         — Group routes by domain
    ↓
services/        — Business logic
    ↓
db/              — All DB queries, nothing else
    ↓
Supabase
```

Each layer only talks to the layer directly below it. Frontend never touches DB. Routes never write SQL. DB layer never has business logic.

---

## Folder Structure

```
backend/
├── main.py                        — App entry, registers all routers
├── routers/
│   ├── chat.py                    — POST /api/chat
│   ├── agents.py                  — CRUD /api/agents
│   ├── documents.py               — POST /api/ingest, GET /api/documents
│   └── users.py                   — POST /api/users, auth routes
├── services/
│   ├── chat_service.py            — orchestrates agents for chat
│   ├── ingestion_service.py       — orchestrates ingestion pipeline
│   └── agent_service.py           — business logic for agent management
├── db/
│   ├── supabase_client.py         — single Supabase connection
│   ├── agents_db.py               — all SQL for agents table
│   ├── documents_db.py            — all SQL for documents table
│   ├── chunks_db.py               — all SQL for parent/child chunks
│   └── users_db.py                — all SQL for users table
├── agent_1_intent_classifier.py
├── agent_2_query_orchestrator.py
└── ...
```

## How Data Flows — Example: User sends a chat message

```
POST /api/chat
    ↓
routers/chat.py        — validates request, calls service
    ↓
services/chat_service.py   — calls classifier, orchestrator, retrieval, synthesizer
    ↓
db/chunks_db.py        — vector search query to Supabase
    ↓
Supabase
```

```python
from fastapi import FastAPI

app = FastAPI()

app.include_router(agents.router,    prefix="/api")
app.include_router(documents.router, prefix="/api")
app.include_router(users.router,     prefix="/api")
```

`main.py` stays 10 lines forever, no matter how big the app grows.

| Layer | Allowed to do | Not allowed to do |

What was this issue?

---


Pydantic (the validation library FastAPI uses) is strict about types.

You had:
```python
company_name: str = None
website: str = None
```

This tells Pydantic: "this field is a `str`, but defaults to `None`". When the frontend actually sends `null`, Pydantic sees it and says — "you told me this is a string, but you gave me null, that's invalid" → 422 error.

The fix `Optional[str] = None` tells Pydantic: "this field can be either a `str` **or** `None`" — so both cases are valid.

`Optional[str]` is just Python shorthand for `Union[str, None]`.

What are different types of error. Can you list all

---


Here are the HTTP error codes you'll commonly encounter:

**2xx — Success**
| Code | Meaning |
|---|---|
| 200 | OK — request succeeded |
| 201 | Created — resource was created (use this for signup) |
| 204 | No Content — success but nothing to return (use for delete) |

**4xx — Client errors (your fault — bad request)**
| Code | Meaning |
|---|---|
| 400 | Bad Request — invalid data sent |
| 401 | Unauthorized — not logged in / wrong credentials |
| 403 | Forbidden — logged in but no permission |
| 404 | Not Found — resource doesn't exist |
| 409 | Conflict — duplicate (username/email already taken) |
| 422 | Unprocessable Entity — data format is wrong (Pydantic validation failure) |
| 429 | Too Many Requests — rate limited |

**5xx — Server errors (backend's fault)**
| Code | Meaning |
|---|---|
| 500 | Internal Server Error — something crashed in backend |
| 502 | Bad Gateway — backend is down or unreachable |
| 503 | Service Unavailable — server overloaded or deploying |
| 504 | Gateway Timeout — backend took too long to respond |

---

**In your specific case:**
- `422` — Pydantic rejected the request shape (what just happened)
- `400` — you should use this for "username taken" or "email already registered"
- `401` — wrong password on login
- `500` — your backend crashed (check Railway logs)

**Quick rule:**
- `4xx` = fix your request
- `5xx` = fix your server
